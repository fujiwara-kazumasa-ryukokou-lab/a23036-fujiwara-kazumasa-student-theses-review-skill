# 学生 issue コメントへの対処

指導者が起票した issue に学生がコメントしてくることがある。更新レビューと**同じセッションで必ず確認**する。

## 学生コメントの典型パターン

| パターン | 例 | エージェントの対処 |
|---------|-----|-------------------|
| 修正報告 | 「対応しました」「直しました」 | リポの最新コミット・該当ファイルを**実際に確認**してから返信 |
| 質問 | 「こういう理解で合っていますか」 | 仕様・意図を簡潔に回答。曖昧なら確認事項を列挙 |
| 反論・理由説明 | 「意図的にこうしています」 | 設計意図を読み、妥当なら issue を更新または close 提案。不十分なら残課題を明記 |
| 部分対応 | 「一部だけ直しました」 | 受け入れ条件をチェックし、未達項目をコメント |
| ブロッカー | 「分からない」「環境がない」 | 次の具体的手順を1〜3ステップで提示 |

## 手順

### 1. 未対応コメントの検出

`run-review.sh` の `pending_issues` を使う（推奨）。単体実行する場合:

```bash
bash skills/student-theses-review/scripts/list-pending-issue-responses.sh --json
```


`pending_issues` が空でなければ対処する。

### 2. 内容の検証（必須）

学生の「直しました」は**信用せず**、次を実行する。

1. `gh issue view <N> --repo <ORG>/<REPO> --comments` で全文を読む
2. 該当リポを fetch 済みであることを確認
3. 学生が示したコミット SHA・ファイルパスを `git show` / `Read` で検証
4. issue の受け入れ条件と照合

### 3. 返信（`gh issue comment`）

返信は**日本語**、**敬体**、**短く具体的**に。

**修正が確認できた場合**（例）:

```markdown
## 確認結果（指導者レビュー）

コミット `<sha>` を確認しました。以下は充足しています。

- [x] …

残り: …（あれば）

受け入れ条件を充足しているため close します（[issue-close-policy.md](issue-close-policy.md)）。
```

**未達の場合**（例）:

```markdown
## 確認結果（指導者レビュー）

コミット `<sha>` を確認しましたが、受け入れ条件のうち次が未達です。

- [ ] …

上記対応後、再度コメントいただくか push をお願いします。
```

### 4. issue 状態の更新

| 状況 | 操作 |
|------|------|
| 全受け入れ条件を満たした | 確認コメントのうえ **`gh issue close` を実行**（[issue-close-policy.md](issue-close-policy.md)） |
| 一部のみ充足 | issue は open のまま、コメントで未達を明記 |
| 設計上 issue 自体が不要と判明 | 理由をコメントし、指導者判断で close |
| 新たな問題が見つかった | 同一論点ならコメント、別論点なら新規 issue |
| **学生が指導者起票 issue を close した** | `gh issue reopen` + 理由コメント。恒久対策は [issue-close-policy.md](issue-close-policy.md) の Actions ガード |

### 5. 禁止事項

- 学生コメントを読まずに「了解」だけ返す
- 検証なしで issue を close する
- **学生に issue close をさせる**（修正報告はコメントのみ）
- commit / PR に `Closes #N` を書く（学生への指示時も同様に注意）
- 同一内容の返信を重複投稿する

## 指導者アカウントの既定

環境変数 `SUPERVISOR_LOGINS`（カンマ区切り）で指定。既定: `KazumasaFUJIWARA`

このログイン以外のコメントを「学生・外部」とみなす（bot は除外）。
